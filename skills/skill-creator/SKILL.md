# Skill Creator — мета-скилл для создания скиллов

## Когда использовать

- Замечена повторяющаяся инструкция (2+ раза) → предложи скилл
- Пользователь просит «сделай скилл», «оформи как скилл»
- Конец сессии → проверь, что забрать в скилл навсегда (Правило 4)
- Существующий скилл разросся → рефакторинг на 2-3 узких (Правило 3)

## Принципы (из CLAUDE.md)

1. **Три слоя, а не один промпт** — Description + Instructions + Tools
2. **Композиция, не монолит** — 3-5 фокус-скиллов, каждый делает ОДНО
3. **Код > генерация** — повторяемая логика в `tools/`, творческая часть — модель
4. **Description решает всё** — расплывчатое описание = скилл не подцепится

## Инструкции

### Шаг 1: Квалификация — это скилл или нет?

Проверь:
- [ ] Задача повторялась 2+ раза? → ДА = скилл
- [ ] Есть детерминированная часть (можно скрипт)? → вынеси в `tools/`
- [ ] Задача одна и конкретная? → ДА. Если несколько → разбей на скиллы
- [ ] Не дублирует существующий скилл? → проверь `ls ~/.claude/skills/`

Если НЕ скилл (разовая задача) — не создавай. Скажи пользователю.

### Шаг 2: Определи имя и description

**Имя:** kebab-case, 2-3 слова, глагол или существительное
- Хорошо: `unit-economics`, `site-from-ref`, `excel-model`
- Плохо: `helper`, `utils`, `do-stuff`, `content-creation`

**Description:** одно предложение, отвечает на «КОГДА брать этот скилл»
- Хорошо: `"Когда пользователь даёт ссылку на Framer/Dribbble и просит сделать похожий сайт"`
- Плохо: `"Помогает с сайтами"` (слишком размыто, не активируется)

Формула: `"Когда [триггер пользователя] — [что скилл делает]"`

### Шаг 3: Создай структуру

```bash
mkdir -p ~/.claude/skills/<имя>/tools ~/.claude/skills/<имя>/examples
```

### Шаг 4: Напиши SKILL.md (слои 1 + 2)

```markdown
---
name: <kebab-case-имя>
description: "<когда брать этот скилл — конкретно>"
---

# <Название> — <что делает в 5 словах>

## Когда использовать
- Триггер 1 (что скажет/сделает пользователь)
- Триггер 2
- Триггер 3

## Инструкции
### Шаг 1: <Что собрать/определить>
### Шаг 2: <Что запустить/сделать>
### Шаг 3: <Как интерпретировать/выдать результат>

## Принципы
- <Правило 1>
- <Правило 2>
```

Правила для SKILL.md:
- Максимум 100 строк (длиннее = скилл перегружен → разбей)
- Шаги нумерованы, каждый — одно действие
- Нет общих фраз типа «учитывай контекст» — только конкретика

### Шаг 5: Заполни tools/ (слой 3)

Вопрос для каждой задачи внутри скилла:
> «Эту часть можно сделать скриптом — стабильно, 0 токенов?»

Если ДА → пиши скрипт (Python/Bash/JS)
Если НЕТ (творческая/контекстная) → оставь модели

Типы файлов в tools/:
| Файл | Когда |
|------|-------|
| `*.py` | Вычисления, генерация файлов, парсинг |
| `*.sh` | CLI-команды, деплой, git-операции |
| `*.md` | Чек-листы, шпаргалки, промпты для WebFetch |
| `*.css` | Библиотеки стилей для copy-paste |
| `*.html` | Шаблоны с {{переменными}} |
| `*.json` | Конфиги, словари, справочники |

Правила для скриптов:
- Принимают аргументы через `argparse` (Python) или `$1 $2` (Bash)
- Выводят результат в stdout (человекочитаемо)
- Работают без установки зависимостей (или ставят сами через pip)
- Протестированы сразу после создания

### Шаг 6: Добавь examples/ (few-shot память)

Минимум 1 пример — реальный кейс из работы с пользователем:
```markdown
# Пример: <название кейса>

## Задача
<Что попросил пользователь>

## Входные данные
<Параметры>

## Команда
<Как запускал скрипт>

## Результат
<Что получилось>

## Заметки
<Что важно запомнить>
```

### Шаг 7: Протестируй

- [ ] Скрипт запускается без ошибок
- [ ] Description точно описывает когда брать
- [ ] SKILL.md < 100 строк
- [ ] tools/ не пустой
- [ ] examples/ есть хотя бы 1

### Шаг 8: Подтверди у пользователя

Покажи:
1. Имя и description
2. Структуру папки
3. Результат тестового запуска (если есть скрипт)

## Антипаттерны (НЕ делать)

| Антипаттерн | Почему плохо | Что делать |
|-------------|-------------|-----------|
| Скилл-монолит (всё в одном) | Ломается, не тестируется | Разбей на 3-5 узких |
| Пустой tools/ | 90% силы скилла — в инструментах | Вынеси логику в скрипт |
| Размытый description | Не активируется автоматически | «Когда [триггер]» |
| Скилл на разовую задачу | Захламляет, не переиспользуется | Не создавай |
| Дублирование скилла | Путаница, какой брать | Расширь существующий |
| SKILL.md > 100 строк | Перегружен, пора разбивать | Рефакторинг |
| Скрипт без argparse | Нельзя менять параметры | Добавь аргументы |
| Нет примера | Следующая сессия не знает как было | Добавь 1 реальный кейс |

## Рефакторинг существующего скилла

Триггеры для рефакторинга:
- SKILL.md > 100 строк
- tools/ > 5 файлов
- Скилл делает 3+ разных вещи
- Пользователь жалуется на результат

Действия:
1. Определи 2-3 чёткие зоны ответственности
2. Создай новые узкие скиллы
3. Перенеси tools/ и examples/ в соответствующие
4. Удали старый скилл
5. Протестируй каждый новый отдельно

## Конец сессии — чек Правила 4

Задай себе:
> «Что из этой сессии забрать в скилл навсегда, а что было разовой правкой?»

1. Просмотри переписку после применения скилла
2. Найди, что пользователь правил руками или объяснял повторно
3. Предложи конкретные правки в SKILL.md или tools/
4. Внеси принятое
